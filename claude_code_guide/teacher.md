# Claude Code 教案提纲

> 面向组内程序员，按图片顺序讲解，口语化介绍。

---

## 开场白

今天给大家介绍一下 Claude Code，我们组已经用了一段时间了，整体感受是——它不只是个代码补全工具，更像一个能真正理解项目背景的协作者。我准备了一系列图，顺着讲下来，大家就能明白怎么把它用好。

---

## 总览图：完整学习路径

**一句话**：四个阶段，从工具使用到底层原理。

这张图是整个分享的地图，先看一眼，知道我们要走哪条路。

四个阶段从上到下：

- **阶段1：基础认知**——最小上下文集合、工作流、Agent Loop，理解 Claude Code 怎么运作
- **阶段2：知识体系**——上排三个（知识架构、Context 用量、知识飞轮）+ 下排四个（CLAUDE.md、Skills、Memory、Subagents），三层协作让 Claude 越用越懂你的项目
- **阶段3：实战方法**——Plan Mode 和有效对话，把工具用对
- **阶段4：提示词工程**——K-shot、CoT、Self-Consistency、RAG、Reflexion，理解底层原理，写出更好的 CLAUDE.md 和 Skills

**要点**：前三个阶段讲"怎么用"，第四个阶段讲"为什么这样用"。

---

## 图1：最小上下文集合

**一句话**：给 Claude 的信息不是越多越好。

这张图是个钟形曲线。左边是信息太少，Claude 不知道你在干什么；右边是信息太多，一堆无关的东西反而干扰它推理；中间那个峰值就是"最小必要上下文"。

对我们项目来说，最小集合就是：M100 硬件架构 + 精度校验方法 + 编译工作流 + 同步机制。把这些告诉它，它就能高效工作。

**要点**：精准 > 全量堆砌。

---

## 图2：工作流

**一句话**：从问题到解决，Claude 有一套完整的路径。

六个阶段：问题输入 → 上下文收集 → Plan 模式规划 → 执行 → 验证 → 结果。

重点说一下执行阶段，有三种模式：
- **Auto Edit**：谨慎，每步确认
- **Plan 模式**：推荐，先规划再动手
- **Yolo 模式**：高效但高风险，适合你完全清楚在做什么的时候

验证这步很关键，我们项目要求 `python3 compare.py` bit-wise 对齐才算成功，不能只看编译通过。

---

## 图3：Agent Loop

**一句话**：Claude 不是一问一答，它是个循环执行的 Agent。

这张图展示了 Claude 内部的执行机制：构建对话 → LLM 推理 → 解析响应 → 判断是否有工具调用 → 执行工具 → 结果追加回对话 → 再次推理……循环直到给出最终答案。

理解这个很重要，因为它解释了为什么 Claude 能自主完成多步骤任务，也解释了为什么 context 会越来越大——每次工具调用的结果都会追加进去。

**要点**：每次调用前 system_prompt 都会重新拼接，所以 CLAUDE.md 的内容每次都在。

---

## 图4：知识架构

**一句话**：Claude 的"知识"来自三个地方。

三层体系：
- **CLAUDE.md**：项目宪法，每次对话都加载，告诉 Claude 这个项目是什么、规则是什么
- **MEMORY.md**：跨会话记忆，Claude 自己写的，记录它学到的经验
- **Skills**：专项知识库，按需加载，只在需要的时候才进 context

这三层加在一起，就是 Claude 对你项目的全部认知。

---

## 图5：Context 用量

**一句话**：200k token 是有限资源，要主动管理。

`/context` 命令可以看当前用量。典型分布：
- System prompt（CLAUDE.md 等）：~14%
- System tools（工具描述）：~11%
- Messages（对话历史）：~29%
- 剩余空间：~26%

Messages 是最大消耗者。读大文件、长对话、工具结果都会快速消耗。Free 不足时 Claude 会开始遗忘早期对话，性能下降。

**要点**：定期 `/context` 检查，新任务开新会话。

---

## 图6：知识飞轮

**一句话**：Claude 越用越强，不是比喻，是真的。

飞轮四个节点：遇到问题 → 召唤知识 → 解决问题 → 沉淀知识 → 下次更快。

实际效果：同样的 HDMA 越界问题，第一次从零摸索要 3 小时，把根因写进 MEMORY.md 之后，第二次 10 分钟搞定。

**要点**：解决完问题记得让 Claude 把经验写进 MEMORY.md，这是飞轮加速的关键动作。

---

## 图7：CLAUDE.md 怎么写

**一句话**：CLAUDE.md 是索引，不是百科全书。

ETH Zurich 有个研究结论：上下文文件写得太冗余，反而会降低解决率约 3%，同时成本增加 20%。所以精简才是关键。

五个核心模块：
1. **Identity**：告诉 Claude 它是谁、这个项目是什么
2. **Output**：输出契约，比如"修改代码后必须给验证命令"
3. **Quality**：质量标准，比如"bit-wise 对齐才算成功"
4. **Boundaries**：禁止操作，比如"不能 git push"
5. **Navigation**：关键文件路径 + Skills 索引

用 `@file` 引用把细节拆到单独文件，CLAUDE.md 只保留索引。只写反复出错的点，不写显而易见的东西。

---

## 图8：Skills

**一句话**：Skills 是按需加载的知识模块，解决 CLAUDE.md 塞不下的问题。

为什么需要 Skills？CLAUDE.md 全量加载，如果把所有知识都塞进去，token 浪费严重。Skills 只在触发时才进 context。

激活流程：用户提问 → Claude 看 Skills 列表的 description → 判断是否需要加载 → 加载完整 Skill 内容 → 回答。

**关键**：description 写得好不好，决定 Skill 能不能被正确触发。内容写得再好，description 不准确也没用。

Skills 可以组合使用，也可以跨项目复用。

---

## 图9：Memory

**一句话**：CLAUDE.md 是你写的，MEMORY.md 是 Claude 自己写的。

什么值得写进 MEMORY.md：解决过的 bug 根因、硬件坑点、跨会话稳定的模式、用户偏好。

什么不该写：当前任务临时状态、未验证的猜测、和 CLAUDE.md 重复的内容。

**要点**：MEMORY.md 超过 200 行会被截断，保持精简，详细内容放主题文件，MEMORY.md 只做索引。

---

## 图10：Subagents

**一句话**：让 Claude 分身，保持主对话干净。

Subagent 在独立的 context 里工作，只把摘要返回给主对话。好处是主 context 不被大量文件内容污染，还能并行执行多个调查任务。

三个内置 Subagent：
- **Explore**（Haiku，只读）：代码搜索，便宜快速
- **Plan**（只读）：Plan Mode 里的研究阶段
- **General-purpose**（全部工具）：复杂多步骤任务

什么时候用 Subagent：任务会产生大量输出、需要限制只读权限、任务相对独立。什么时候用主对话：需要频繁迭代、多阶段共享状态。

## 图11：Transition（工具齐了，怎么用？）

**一句话**：前面讲的都是装备，接下来讲战术。

这张图是个过渡，总结四个核心认知：Context 是最重要的资源、工具让 Claude 越来越懂你的项目、Subagents 隔离高消耗操作、飞轮越转越快。然后引出接下来两张：Plan Mode 和有效沟通。

---

## 图12：Plan Mode

**一句话**：先想清楚再动手，思考和行动解耦。

四个阶段：
1. **探索**（Plan Mode，只读）：读文件、Grep 代码，零副作用
2. **规划**（Plan Mode）：生成可审核的分步计划，列出要改哪些文件、每步理由
3. **人工审批**：`Shift+Tab` 进入，循环 review → 修改 → re-review，直到满意
4. **实现**（Normal Mode）：按计划编码，边写边验证

**为什么重要**：你可以在它动手之前看到完整计划，发现问题直接改计划，不用等它写完代码再回滚。

---

## 图13：有效对话与会话管理

**一句话**：沟通方式决定结果质量。

六个策略：

1. **直接提问**：像问资深工程师一样问，不需要特殊提示格式。
2. **让 Claude 采访你**：大功能开发前，让 Claude 用 AskUserQuestion 采访你，它会问你没想到的边界情况。采访完写进 SPEC.md，然后开新会话执行。
3. **尽早纠偏**：发现跑偏立刻 `Esc` 打断。同一问题纠偏超过两次，直接 `/clear` 重新开始，带上你学到的东西。
4. **管理 Context**：`/clear` 完全重置、`/compact` 压缩保留关键、`/btw` 快速问答不进历史、`Esc×2` 从某条消息总结。
5. **Subagents 调查**：委托研究用"use subagents to investigate X"，主对话保持干净。
6. **恢复会话**：`claude --continue` 继续最近会话，`claude --resume` 选择历史，`/rename` 给会话起名，像管理分支一样管理会话。

## 图14：Transition（从使用到原理）

**一句话**：前面讲的是"怎么用"，现在讲"为什么这样用"。

这张图是个过渡，左边总结 Claude Code 使用层（知识体系、工作模式、对话管理），右边引出提示词工程基础（5个核心技术）。理解这些底层原理，你就能写出更精准的 CLAUDE.md、更高效的 Skills、更有效的提示词。

五个提示词工程技术：
1. **K-shot 提示**：通过示例学习，Skills 的 description 里带具体例子
2. **CoT 思维链**：逐步推理，Plan Mode 自动使用 CoT
3. **Self-Consistency**：多次投票提升准确率，复杂推理的保障
4. **RAG 检索增强**：基于文档回答，Skills 的实现原理
5. **Reflexion 反思**：错误反馈迭代，Claude 能自己改 bug 的原因

---

## 图15：K-shot 提示 - 示例的质量 > 数量

**一句话**：提示词里加示例，模型就能学会模式。

左右对比：
- **左边（失败）**：0-shot 提示词，只说"Reverse the word"，没有示例。模型不理解具体含义，可能理解成单词顺序反转、大小写转换等，成功率约 10%。
- **右边（成功）**：Few-shot 提示词，给了 3 个示例（http→ptth, status→sutats, teststatus→sutatstset），还说明了过程"Take letters from right to left"。关键是示例结构相似（都是长度 10+ 的复合词），成功率约 90%。

**实战应用**：
- Skills 的 description 里必须带具体例子："分析 M100 同步机制。例如：检查 HDMA 配置"
- CLAUDE.md 里的关键指令配示例："修改代码后必须验证。例如：python3 compare.py"

**提示词模板**：任务描述 + 3 个结构相似的示例 + 过程说明

---

## 图16：CoT 思维链 - 提示词里加"先想再答"

**一句话**：一句话触发推理过程，准确率从 1% 提升到 60%。

左右对比：
- **左边（失败）**：普通提示词"Calculate: 3^12345 mod 100 = ?"，模型直接猜答案 87，没有推理过程，准确率约 1%（100 个数字里随机猜）。
- **右边（成功）**：CoT 提示词，加了"Think step by step"触发词。模型展示完整推理：Step 1 找周期（3^20≡1 mod 100）→ Step 2 计算余数（12345=617×20+5）→ Step 3 简化（3^12345≡3^5）→ Step 4 计算结果（243 mod 100=43）。准确率 60-80%。

**为什么有效**：
- 模型"跳步"会丢失中间逻辑
- 展示推理步骤，像考试写步骤一样
- 能应用数学定理、找到规律

**实战应用**：
- CLAUDE.md 里写："复杂问题先列推理步骤"
- Plan Mode 自动使用 CoT

**提示词模板**："Think step by step" / "Let's work this out step by step"

---

## 图17：Self-Consistency - 提示词 + temperature=1 + 多次运行

**一句话**：同一提示词运行 5 次，投票选最可靠答案。

左右对比：
- **左边（失败）**：temperature=0 单次运行，输出确定但可能错（Answer: 35），准确率约 40%。
- **右边（成功）**：temperature=1 + 运行 5 次。5 次输出：25、25、25、35、25，投票结果 25 得 4 票，35 得 1 票，多数答案 25 正确。准确率 70-85%。

**为什么有效**：
- temperature=1 增加多样性，每次推理路径不同
- 多次运行，即使有 1 次错误，多数投票仍能得到正确答案
- 多样性推理 + 投票机制过滤错误

**实战应用**：
- 复杂推理任务必备
- 代码生成时可以生成多个版本，选最一致的

**提示词配置**：temperature=1（增加多样性）+ NUM_RUNS=5 + 投票选多数

---

## 图18：RAG 检索增强 - 提示词里注入外部文档

**一句话**：Context: {文档} + Task: {问题} → 基于事实回答。

左右对比：
- **左边（失败）**：无 RAG 提示词，只说"写一个 Python 函数调用 API 获取用户名"，没有提供 API 文档。模型编造 URL（http://api.com/user）、缺少认证头，产生幻觉，代码无法运行。
- **右边（成功）**：RAG 提示词，注入了完整 API 文档（Base URL、GET /users/{id}、Header: X-API-Key）到 Context，并加上"use ONLY this information"防止编造。模型基于文档生成正确代码，可直接运行。

**为什么有效**：
- 模型不知道真实 API 文档会编造
- 把外部知识注入到提示词，让模型基于事实回答
- "use ONLY this"防止幻觉

**实战应用**：
- Skills 就是 RAG，把专项文档按需加载到 context
- CLAUDE.md 里用 `@file` 引用外部文档

**提示词模板**：Context: {文档} + "use ONLY this information" + Task: {问题}

---

## 图19：Reflexion 反思 - 提示词里加"自我批判"指令

**一句话**：生成 → 测试反馈 → 反思提示词（分析错误+改进）→ 重新生成。

左右对比：
- **左边（失败）**：简单重试提示词，测试失败后只说"再试一次"，没有分析为什么失败，盲目重试，可能重复错误。
- **右边（成功）**：Reflexion 提示词，包含三部分：
  1. Previous code: {之前的代码}
  2. Test failures: {具体错误信息}（如"缺少大写字母检查"）
  3. "分析并改进"指令

  模型根据具体错误针对性修复，测试通过。

**为什么有效**：
- 提供具体错误信息，而不是盲目重试
- 模型能分析失败原因，针对性改进
- 错误驱动的迭代改进

**实战应用**：
- Claude 能自己改 bug 的原因
- CLAUDE.md 里写："测试失败时先分析原因再修改"

**提示词模板**：Previous: {代码} + Failures: {错误} + "Analyze the failures and improve the code"

---

## 总结

这 19 张图构成了一套完整的使用方法论：

- **基础认知**（图1-3）：最小上下文、工作流、Agent Loop
- **知识体系**（图4-10）：CLAUDE.md + MEMORY.md + Skills，三层协作，飞轮加速，Subagents 隔离
- **过渡1**（图11）：从知识体系到实战方法
- **实战方法**（图12-13）：Plan Mode 结构化工作流 + 有效沟通策略
- **过渡2**（图14）：从实战方法到提示词工程
- **提示词工程**（图15-19）：K-shot、CoT、Self-Consistency、RAG、Reflexion 五大技术

核心就一句话：**精准上下文 + 知识飞轮 + 结构化工作流 + 提示词工程**，Claude 会越用越顺手。

**提示词工程的价值**：理解这 5 个技术，你就能：
- 写出更精准的 CLAUDE.md（用 CoT 要求展示推理、用 K-shot 给示例）
- 写出更高效的 Skills（用 RAG 注入文档、用 K-shot 写 description）
- 写出更有效的提示词（用 Self-Consistency 处理复杂推理、用 Reflexion 处理错误反馈）
